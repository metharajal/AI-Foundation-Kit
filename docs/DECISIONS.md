## Decisions

### DEC-001 — AEOS is a sovereign software continuity platform

**Date:** 2026-06-30
**Status:** Accepted
**Capability:** Govern
**Decided by:** AEOS Constitution Sprint

**Context:**
The original definition "AI Engineering Operating System" captured the tooling aspect but did not capture the mission: ensuring software continues to exist, operate, and remain owned by its builders over time. The term "continuity" expresses the full arc from creation to sovereign operation.

**Options considered:**
- Keep "AI Engineering Operating System" — accurate but incomplete
- "Sovereign software continuity platform" — captures mission and posture
- "Engineering sovereignty platform" — shorter but less precise

**Decision:**
AEOS is both. The primary identity is "sovereign software continuity platform." The secondary identity is "AI Engineering Operating System." Both are true and non-contradictory.

**Rationale:**
The word "continuity" captures what AEOS truly does: it ensures a software project can continue to be owned, operated, and evolved regardless of the platform that created it. "Sovereign" positions the intent. Together they describe the product truthfully.

**Consequences:**
All documentation, tooling, and communications integrate "sovereign software continuity platform" as the primary definition of AEOS.

### DEC-002 — Eight canonical capabilities

**Date:** 2026-06-30
**Status:** Accepted
**Capability:** Govern
**Decided by:** AEOS Constitution Sprint

**Context:**
Prior documentation used "modes," "rails," "use cases," and "features" inconsistently to describe what AEOS does. A canonical capability set was needed to anchor the identity and prevent vocabulary drift.

**Options considered:**
- Define 6 modes matching the product vision sections
- Define 9 capabilities matching the 9 rails
- Define 8 capabilities as a clean abstraction above both

**Decision:**
Eight capabilities: Discover · Assess · Recover · Transform · Continue · Govern · Operate · Learn.

**Rationale:**
The 8 capabilities map cleanly to the 9 rails (some rails combine capabilities) while remaining stable across product evolution. They describe *what AEOS does*, not *how it does it*.

**Consequences:**
New features are described in terms of which capability they implement. The capabilities are constitutional (Part I) and do not change without a constitutional amendment.

### DEC-003 — Ten platform engines

**Date:** 2026-06-30
**Status:** Accepted
**Capability:** Govern
**Decided by:** AEOS Constitution Sprint

**Context:**
The code organized around a set of modules, but these modules were not formally named as "engines" with a documented purpose. The Transformation Engine (migration) was missing from earlier engine lists.

**Options considered:**
- Keep module names without formal engine designations
- Define 8 engines (one per capability)
- Define 10 engines including Memory, Evidence, and AI Routing as first-class

**Decision:**
Ten engines: Discovery · Assessment · Recovery · Transformation · Continuation · Governance · Operations · Memory · Evidence · AI Routing.

**Rationale:**
Memory, Evidence, and AI Routing are cross-cutting concerns that serve all capabilities. Elevating them to first-class engines ensures they are designed and maintained as core infrastructure, not afterthoughts.

**Consequences:**
The Transformation Engine (previously missing) is now explicit. The engine catalog evolves through the Standards process. Adding or removing engines requires a constitutional amendment or a Standards RFC.

### DEC-004 — Three orthogonal level systems

**Date:** 2026-06-30
**Status:** Accepted
**Capability:** Govern
**Decided by:** AEOS Constitution Sprint

**Context:**
AEOS documentation referenced "levels" in three different contexts — action authorization, project maturity, and cloud sovereignty — without clearly distinguishing them. This caused confusion when reading recommendations.

**Options considered:**
- Unify into one level system
- Keep three separate systems, unnamed
- Define three systems with distinct names and purposes

**Decision:**
Three formally named, orthogonal systems:
1. **Action Levels (0–5)** — operation authorization
2. **Project Maturity (weak→sovereign)** — project recovery state
3. **Sovereignty Levels (1–5)** — cloud infrastructure dependency

**Rationale:**
A project can be at Project Maturity "controlled" while remaining at Sovereignty Level 1 (still on Supabase Cloud). These are independent axes. Conflating them produces incorrect recommendations.

**Consequences:**
Every AEOS output that references "level" must specify which system. Documentation distinguishes the three systems explicitly.

### DEC-005 — CLI is the authoritative interface

**Date:** 2026-06-30
**Status:** Accepted
**Capability:** Govern
**Decided by:** AEOS Constitution Sprint

**Context:**
AEOS has four interfaces (CLI, API, Workspace, Agents). As the API and Workspace are built, behavior discrepancies will emerge. The authoritative source of correct behavior needed to be defined.

**Options considered:**
- API as authoritative (common in cloud-first tools)
- All interfaces equal (leads to drift)
- CLI as authoritative (local-first, inspectable, version-controlled)

**Decision:**
CLI is the primary, authoritative interface. When behavior differs between interfaces, CLI defines correct behavior.

**Rationale:**
AEOS is local-first. The CLI is the only interface that exists today, is fully tested, and embodies the invariants (read_only, applied=false, no secret display). Treating it as authoritative keeps future interfaces honest.

**Consequences:**
API and Workspace are implemented to match CLI semantics, not the reverse. CLI tests are the behavioral contract.

### DEC-006 — Python CLI with Typer, ruff + mypy strict + pytest

**Date:** 2026-06-29
**Status:** Accepted
**Capability:** Govern
**Decided by:** Initial AEOS implementation

**Context:**
The choice of language, CLI framework, and quality toolchain was made at project inception.

**Decision:**
- Language: Python 3.11+
- CLI framework: Typer (type-annotated, composable)
- Package management: uv
- Quality: ruff (linting), mypy strict (type checking), pytest (tests)
- Convention: `from __future__ import annotations` in all modules

**Rationale:**
Python is accessible, well-supported for CLI tooling, and compatible with the local AI ecosystem (Ollama, LM Studio). Typer provides type-safe CLI construction. ruff + mypy strict + pytest form a non-negotiable quality gate.

**Consequences:**
No code merges without passing all three gates. `uv sync --extra dev --extra test` is the standard setup command.

### DEC-007 — Local AI first, frontier by exception

**Date:** 2026-06-29
**Status:** Accepted
**Capability:** Govern
**Decided by:** Initial AEOS doctrine

**Context:**
AEOS is a sovereignty platform. Its own AI usage must reflect its doctrine. Using frontier AI by default would contradict the platform's stated values.

**Decision:**
Local AI by default. Frontier AI by exception. `require_human_approval = true` — no silent escalation.

**Rationale:**
AEOS cannot credibly recommend local-first AI to clients while itself defaulting to frontier AI. The routing engine (`src/aeos/ai/router.py`) enforces this at the code level.

**Consequences:**
The `auto` provider mode with `require_human_approval = true` raises an error rather than silently escalating. This is a safety feature, not a limitation.

### DEC-008 — MemoryRecord is a safe snapshot — never contains secrets

**Date:** 2026-06-29
**Status:** Accepted
**Capability:** Learn
**Decided by:** Memory Layer Sprint

**Context:**
The Memory Engine stores audit results and findings. A naive implementation could inadvertently store secret variable values, database credentials, or sensitive business data alongside findings.

**Decision:**
`MemoryRecord` is a safe serializable snapshot. It stores only: project metadata, findings summaries (descriptions, not values), remediation paths, strategic options, and human-validated notes. It never stores secret values.

**Rationale:**
A memory system that stores secrets is a liability. The MemoryRecord design ensures the memory layer is safe to inspect, version-control, and share with other team members.

**Consequences:**
`human_validated: false` is the default until a human explicitly confirms the record. Secret values are stripped before any record is written.

### DEC-009 — EvidenceReport uses caller-declared indices, never auto-detection

**Date:** 2026-06-29
**Status:** Accepted
**Capability:** Assess
**Decided by:** Evidence Engine Sprint

**Context:**
The Evidence Engine tracks which expected evidence items for each recovery stage have been confirmed. An auto-detection approach would infer confirmation from filesystem inspection, which is fragile and potentially incorrect.

**Decision:**
Evidence status is computed from caller-declared confirmed indices only. The CLI receives `--confirmed N` flags and passes indices to `build_evidence_report`. Nothing is auto-detected.

**Rationale:**
Evidence is a human attestation, not a machine inference. The human confirms what they have verified. The engine records it. This keeps the evidence chain clean and auditable.

**Consequences:**
`EvidenceReport` always has `read_only=True` and `applied=False`. Validation blocked reason is explicit and human-readable.

### DEC-010 — AEOS Constitution governs AEOS itself (Sprint G1)

**Date:** 2026-06-30
**Status:** Accepted
**Capability:** Govern
**Decided by:** AEOS Self Governance Sprint

**Context:**
AEOS generates governance documentation for client projects as part of the Recovery arc. But AEOS itself lacked the governance documents it recommends: `ARCHITECTURE.md`, `DECISIONS.md`, `SECURITY.md`, `SOVEREIGNTY.md`, `AI-DEVELOPMENT-POLICY.md`.

**Decision:**
AEOS must comply with its own Constitution before pursuing the commercial roadmap. Sprint G1 creates all required governance documents for AEOS itself, applying the same standards AEOS recommends for client projects.

**Rationale:**
A platform that recommends governance practices must itself follow them. Credibility requires compliance. AEOS eating its own cooking is also the best way to validate that the recommended practices are practical.

**Consequences:**
All five required governance documents exist. The governance infrastructure (`governance/adr/`, `governance/rfc/`, `governance/dec/`, `governance/standards/`, `governance/playbooks/`) is initialized. AEOS is now compliant with CONSTITUTION §5.5.
