# AEOS Reclaim Harden

**Date:** 2026-06-28
**Status:** Feature — Stable MVP
**Sprints:** 3A · 3C
**Commands:**
```
aeos reclaim harden --path <project>
aeos reclaim harden --path <project> --json
aeos reclaim harden --path <project> --output <file>
aeos reclaim harden --path <project> --output <file> --overwrite
```

---

## Summary

`aeos reclaim harden` is the top-level sovereignty audit command in AEOS.
It orchestrates five independent read-only checks in a single pass,
produces a consolidated risk report, and emits prioritized recommendations
and strategic exit options — without connecting to any database, reading
any `.env` file, or modifying any file in the client project.

The command is designed to give a developer a complete picture of how much
control they actually hold over a project generated by a no-code or AI tool.
It turns a silent dependency graph into an explicit, actionable risk map.

---

## 1. The Problem `reclaim harden` Solves

Projects built on Lovable, Bolt, Replit, or similar AI generators often ship
with multiple invisible dependencies: a managed Supabase project the developer
does not own, Vercel as the only deployment target, no Dockerfile, no local
backend, keys committed to Git history.

Running `aeos reclaim inspect` alone identifies generators and maps control.
Running `aeos supabase rls harden` alone analyzes RLS policies.
Neither command correlates these two views into a single risk verdict.

`reclaim harden` fills that gap: it runs every relevant check, aggregates
findings across security, sovereignty, database access control, and secret
exposure, and produces one consolidated status with one ordered action list.

---

## 2. Command Differences

| Command | Scope | Output |
|---|---|---|
| `aeos reclaim inspect` | Generator detection, provider mapping, control map, exit options | Structural portrait of the project |
| `aeos reclaim harden` | All of the above + security check + sovereignty check + Supabase check + RLS harden | Full risk report with cross-domain recommendations |
| `aeos supabase rls harden` | RLS policies in `supabase/migrations/` only | SQL proposal with per-table verdict |

**Rule of thumb:**
- Use `reclaim inspect` to understand the project topology before doing anything.
- Use `reclaim harden` as the canonical audit entry point — it calls `inspect` internally.
- Use `supabase rls harden --output` when you are ready to produce a SQL migration file.

---

## 3. Internal Flow

```
aeos reclaim harden --path <project>
│
├── 1. reclaim inspect          → generators, providers, control map, exit options
├── 2. security check           → secrets in code, unsafe dependencies
├── 3. sovereignty check        → ownership and portability signals
│
├── [if Supabase detected]
│   ├── 4. supabase check       → project config, service role exposure, RLS flags
│   └── [if supabase/migrations/ exists]
│       └── 5. supabase rls harden  → per-table RLS analysis + SQL proposal
│
├── Status aggregation          → OK | WARNING | ERROR
├── Recommendation list         → ordered, cross-domain
└── Exit options                → 5 strategic migration paths
```

All steps run locally. No network call is made. No database connection is opened.
Steps 4 and 5 are conditional: they run only when the corresponding artifacts are
detected in the project directory.

---

## 4. Security Guarantees

These invariants are enforced in code and verified by tests. They cannot be
overridden by any flag or option.

| Invariant | Value |
|---|---|
| `read_only` | `true` — no file in the client project is ever written |
| `applied` | `false` — no SQL migration is applied to any database |
| No `.env` read | Variable names are parsed for detection; values are never read or displayed |
| No secret display | Evidence fields contain only `file:line` references, never variable content |
| No network access | All analysis runs entirely on the local filesystem |
| No database connection | No Supabase URL, API key, or service role key is needed or used |

---

## 5. Status Levels

The command emits a single aggregated status across all sub-checks.

| Status | Meaning | Trigger conditions |
|---|---|---|
| `OK ✓` | No critical or structural risk detected | No generator detected · portability strong · no security errors · no secret exposure |
| `WARNING ⚠` | Risks exist but are not immediately blocking | Any generator detected · weak/partial portability · security warnings · RLS WARNING verdict |
| `ERROR ✗` | Critical risk — immediate action required | Secret exposure confirmed or at risk · critical security finding · Supabase CRITICAL/ERROR · RLS BLOCKED verdict |

Exit code: `0` for OK and WARNING · `1` for ERROR.

Each sub-check also carries its own status independently:

| Sub-check | Possible values |
|---|---|
| `security_status` | OK · WARNING · ERROR |
| `sovereignty_status` | OK · WARNING · ERROR |
| `supabase_status` | OK · WARNING · ERROR · CRITICAL · (null if Supabase not detected) |
| `rls_verdict` | PASS · WARNING · BLOCKED · (null if no migrations directory) |

---

## 6. Action Categories

Findings across all sub-checks are grouped into four categories in the report:

### Must fix now

Triggered when `status = ERROR`. Examples:
- Keys in Git history (`secrets_exposure: confirmed`) → rotate all exposed credentials immediately.
- Critical security finding (hardcoded token, unsafe eval) → fix before any deployment.
- Supabase CRITICAL status → service role key exposed in client-side code.
- RLS BLOCKED verdict → migration references a secret or a dangerous pattern.

### Should fix next

Triggered when `status = WARNING`. Examples:
- Any no-code generator detected (Lovable, Bolt, Replit) → project sovereignty is partial.
- Weak portability (no Dockerfile, no local backend, no local schema) → migration is difficult.
- RLS WARNING verdict → TODO blocks need human review before applying.

### Manual review required

These are surfaced as `manual_actions` in the summary. Examples:
- RLS TODO blocks in the generated SQL → require human judgment before use.
- Manual Supabase remediation steps from `aeos supabase check` → e.g., enabling RLS on a table via the Dashboard.

### Strategic migration options

These are the five exit options (see Section 7). They are always emitted regardless of status.

---

## 7. Exit Options

`reclaim harden` always emits five strategic options ordered from least to most disruptive.
Each option carries a complexity and a sovereignty level.

| # | Option | Complexity | Sovereignty |
|---|---|---|---|
| 1 | Stay on current provider but secure | low | partial |
| 2 | Migrate to own Supabase Cloud project | medium | medium |
| 3 | Migrate to self-hosted Supabase | high | high |
| 4 | Migrate to PostgreSQL + open backend | very_high | very_high |
| 5 | Full sovereign rebuild | extreme | maximum |

**Option 1 — Stay on current provider but secure**

Advantages: no migration required, immediate improvement, zero infrastructure cost.
Risks: continued vendor lock-in, keys must be rotated manually.
Next action (when secrets exposure is confirmed): `Rotate exposed keys · run aeos supabase check --path .`

**Option 2 — Migrate to own Supabase Cloud project**

Advantages: you own the project, full Dashboard access, minimal migration effort.
Risks: still cloud-dependent, direct billing, data export required.
Next action: export schema → create new Supabase project → update `.env`.

**Option 3 — Migrate to self-hosted Supabase**

Advantages: full infrastructure control, same Supabase API surface, data on-premises.
Risks: operational burden, no managed Dashboard, updates and backups are your responsibility.
Next action: add `docker-compose.yml` with Supabase stack → migrate data.

**Option 4 — Migrate to PostgreSQL + open backend**

Advantages: zero Supabase dependency, standard open-source stack, maximum portability.
Risks: API layer rewrite required, auth migration required, significant engineering effort.
Next action: extract SQL schema → write API layer → replace Supabase client calls.

**Option 5 — Full sovereign rebuild**

Advantages: total control, local-first from day zero, zero legacy debt.
Risks: maximum cost and delay, UX gap during rebuild.
Next action: define target architecture → `aeos init` → progressive migration.

## 8. Output Modes

### Text (default)

```
aeos reclaim harden --path <project>
```

Emits a structured human-readable report with sections:
- Summary table (generator, providers, control level, secrets exposure, per-check status)
- Critical Risks (if any)
- Important Risks (if any, up to 10 shown)
- Generatable Fixes (SQL blocks ready to export)
- Manual Actions Required (TODO blocks, Supabase steps)
- Recommendations (ordered list)
- Exit Options (5 paths)
- Invariant footer: `read_only: true · applied: false`

### Markdown file export (`--output`)

```
aeos reclaim harden --path <project> --output <file>
aeos reclaim harden --path <project> --output <file> --overwrite
```

Writes a structured Markdown report to `<file>` and prints a short summary to stdout.

**Report structure:**

- AEOS header with `Generated by`, date, project path, `read_only: true`, `applied: false`
- Global status (`OK ✓` / `WARNING ⚠` / `ERROR ✗`)
- Summary table (all sub-check statuses and counts)
- Critical Risks section (if any `ERROR` findings)
- Important Risks section (if any `WARNING` findings)
- Generatable Fixes section (if auto-generated SQL blocks exist)
- Manual Actions Required section (RLS TODO blocks, Supabase manual steps)
- Recommendations (ordered list)
- Exit Options (5 strategic paths)
- Footer: `No correction has been applied.` with all four invariants

**Stdout summary:**

```
Status:           ERROR ✗
Exported:         /tmp/ma-mairie-digitale-reclaim-report.md
Critical risks:   3
Manual actions:   15
Generatable SQL:  25 block(s)
Read-only — no files modified, no migration applied.
  read_only: true  ·  applied: false
```

**Overwrite behavior:**

| Condition | Result |
|---|---|
| File does not exist | Write file ✅ |
| File exists, no `--overwrite` | Refuse — exit 1 ✗ |
| File exists, `--overwrite` given | Overwrite ✅ |

### JSON

```
aeos reclaim harden --path <project> --json
aeos reclaim harden --path <project> --output <file> --json
```

Emits a machine-readable JSON object. Useful for CI pipelines or scripted analysis.

## 9. Validated Case: ma-mairie-digitale

This is the reference validation run for Sprint 3A. The project is a Lovable-generated
civic platform using Supabase for database and authentication.

**Detection results:**

| Signal | Value |
|---|---|
| Generator detected | lovable |
| Providers detected | supabase |
| Control level | weak |

**Sub-check results:**

| Sub-check | Status |
|---|---|
| security | OK ✓ |
| sovereignty | WARNING ⚠ |
| supabase | ERROR ✗ |
| RLS verdict | WARNING ⚠ |

**Secrets exposure:** `confirmed` — `.env` file found in Git history.
This is the primary blocker. All credentials committed to history must be considered
compromised and rotated before any other action.

**Invariants verified on this run:**

```
read_only: true
applied:   false
```

No file in the `ma-mairie-digitale` directory was modified.
No secret value was read, stored, or displayed.
No database connection was opened.

**Key findings on this run:**
- Lovable detected via `src/integrations/supabase/` directory structure.
- Supabase detected via Supabase client dependency in `package.json`.
- No Dockerfile, no local backend directory, no local schema directory → portability: weak.
- `.env` found in Git history → `secrets_exposure: confirmed` → overall status: ERROR.
- Supabase check returned ERROR due to security configuration issues.
- RLS harden returned WARNING → auto-generated SQL blocks available, but TODO blocks need review.

**Recommended next actions produced by this run:**
1. Rotate all exposed keys immediately — they are in Git history.
2. Review RLS TODO blocks manually before applying any migration.
3. Export auto-generated RLS blocks with `aeos supabase rls harden --output`.
4. Complete manual Supabase remediation steps from `aeos supabase check`.
5. Consider adding Dockerfile and local backend to improve portability.

## 10. Current Limits

| Limit | Detail |
|---|---|
| No apply gate | `reclaim harden` cannot apply fixes. It only produces a report and a SQL proposal via `supabase rls harden --output`. Applying migrations is a deliberate human action. |
| No CI integration output | JSON output is machine-readable but there is no native GitHub Actions annotation format. |
| Single-provider RLS | RLS harden only targets Supabase PostgreSQL migrations. Firebase, PlanetScale, and other providers are detected but not analyzed at the SQL level. |
| Git history scan is shallow | Secret exposure detection uses `git log --oneline -- .env`. It detects presence, not content. |
| Exit options are static | The five paths are always emitted with fixed complexity/sovereignty ratings. They do not yet adapt dynamically to the detected provider stack. |
| Remediation plan is rule-based | Phase priorities and actions are derived from audit findings deterministically. No AI inference is used at this stage. |

## 11. Remediation Plan

`reclaim harden` now includes a 5-phase remediation plan built from audit findings.

### Phases

| Phase | Label | Priority | Automation |
|---|---|---|---|
| Phase 0 | Immediate security stabilization | `critical` if secrets exposed · `medium` otherwise | manual |
| Phase 1 | Database and RLS hardening | `critical` if BLOCKED · `high` if WARNING | `generatable` if auto blocks exist |
| Phase 2 | Application control recovery | `high` if generator detected · `medium` otherwise | assisted |
| Phase 3 | Portability preparation | `medium` if portability weak/partial · `low` otherwise | assisted |
| Phase 4 | Strategic exit path | `low` always | manual |

### Phase content

Each phase includes:
- `priority` — `critical` · `high` · `medium` · `low`
- `goal` — what this phase achieves
- `why_it_matters` — consequence of not doing it
- `actions` — concrete ordered steps, tailored to the detected project state
- `automation_level` — `manual` · `assisted` · `generatable`
- `expected_outcome` — observable result when the phase is complete
- `risk_if_skipped` — what happens if this phase is skipped

### Text output summary

```
── Remediation Plan ──────────────────────────────────
  5 phases · 3 immediate · 8 manual · 25 generatable · 5 strategic paths
  ✗ [critical ] phase_0 — Immediate security stabilization
  ⚠ [high     ] phase_1 — Database and RLS hardening
  ⚠ [high     ] phase_2 — Application control recovery
  · [medium   ] phase_3 — Portability preparation
  · [low      ] phase_4 — Strategic exit path
  → Run with --output to export the full plan.
```

### Markdown export

`--output <file>` includes the full `## Remediation Plan` section with all 5 phases.

### JSON schema

```json
"remediation_plan": {
  "phases_count": 5,
  "immediate_actions_count": 3,
  "manual_actions_count": 8,
  "generatable_actions_count": 25,
  "strategic_options_count": 5,
  "phases": [
    {
      "id": "phase_0",
      "label": "Immediate security stabilization",
      "priority": "critical",
      "goal": "...",
      "actions": ["...", "..."],
      "automation_level": "manual",
      "expected_outcome": "...",
      "risk_if_skipped": "..."
    }
  ]
}
```

### Invariants

The remediation plan is **diagnostic only**:
- `read_only: true` — no phase applies any fix
- `applied: false` — no migration is run
- No secret value appears in any phase action
- No database connection is made
